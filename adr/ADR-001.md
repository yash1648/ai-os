## Context

AI-OS is a multi-layer system spanning kernel (scheduling, state, diff), knowledge
management (graph index / PIL), planning, and worker execution. The first sprint
(Stage 1) needed to establish:

- Which language(s) for which layer.
- How cross-layer contracts are specified and enforced.
- How the kernel models objectives, manifests, and state transitions.
- How schemas remain the canonical source of truth across Rust, Python, and JSON
  interchange boundaries.

Key constraint: the project must support a polyglot future (Rust for the kernel,
Python for AI-assisted planning, JSON for web/CLI interchange) without allowing
those boundaries to introduce silent drift.

## Decision

### 1. Rust Kernel with sqlx + serde

The kernel (`kernel/`) is implemented in Rust using:
- **sqlx** for compile-time-checked SQLite queries — SQL schema drift is caught
  at compile time, not at runtime.
- **serde** (with `#[serde(rename_all = "snake_case")]`) for JSON serialization
  of all domain types (Objective, Manifest, Event, etc.).
- The terminal `ObjectiveState::Done` label is serialized as `"DONE"` via
  custom `Serialize`/`Deserialize` derives to match the JSON schema.

### 2. JSON Schemas as Canonical Truth

All cross-layer data contracts live as standalone JSON Schema files in
`schemas/`. The Rust serde definitions and (future) Python pydantic models are
*derived* from these schemas, not the other way around. When a mismatch is
detected (as occurred between `schemas/objective.json` defining 5 Priority
variants and the kernel enum only having 4), the schema wins and the code is
corrected.

This ensures the JSON wire format is the authoritative interface definition
language, not an afterthought.

### 3. Python Planner for Stage 2+

The planner layer (`planner/`) uses Python, chosen for:
- Native LLM/AI library ecosystem (langchain, instructor, pydantic).
- Fast iteration speed for the AI-assisted decomposition workflow (Stage 2).
- Pydantic v2 models that can import the JSON schemas directly via
  `pydantic.TypeAdapter` — future-proofing schema enforcement on the planner
  side.

### 4. State Machine with Explicit Transitions

Objective lifecycle follows a finite state machine (docs/07-state-machine.md)
that is mechanically enforced by the kernel. Transitions are table-driven;
illegal transitions (skip, backward, terminal re-entry) are rejected before any
side effects. This prevents objectives from reaching inconsistent states
regardless of caller.

### 5. Diff Applier for Atomic Changes

All kernel-initiated file modifications go through the `DiffApplier` — a
snapshot-first, apply-second, rollback-on-failure primitive. It writes to a
temporary directory, verifies the diff matches the expected current content,
and only then renames into place. This makes the kernel's write path atomic
with respect to failures.

## Alternatives Considered

- **Single-language (all Rust)**: Rejected because AI/LLM tooling in Rust is
  immature and slow to iterate — Python is the stronger choice for the
  planning/GAI layer.
- **Single-language (all Python)**: Rejected because the kernel requires
  strong concurrency guarantees, compile-time SQL checking, and predictable
  performance — Rust's type system and sqlx provide these natively.
- **No JSON Schemas (Rust-only serde)**: Rejected because JSON schemas serve
  as the human-readable, language-agnostic contract that both Rust and Python
  sides reference — removing them would re-introduce drift risk.
- **Mutable diff approach (in-place edits)**: Rejected because in-place
  modification without snapshot+rollback makes the kernel non-atomic — a
  crash during diff application leaves the filesystem in an inconsistent state.

## Rationale

The Rust kernel + JSON schemas + Python planner split matches the natural
competency boundaries of the project:
- Kernel: correctness-critical, concurrent, IO-bound → Rust.
- Schemas: language-agnostic, versioned, reviewable → JSON Schema.
- Planner: AI-heavy, experimental, fast-moving → Python.

The DiffApplier's snapshot-first semantics are derived from the project's
failure philosophy (docs/02-architecture-overview.md — "design for partial
failure"). The state machine's explicit transitions prevent the most common
class of bugs in engineering-workflow trackers: inconsistent state.

## Consequences

### Positive
- Compile-time SQL checking prevents column/type drift.
- JSON schemas serve as executable documentation; any language can implement
  against them.
- DiffApplier makes write failures safe by construction.
- State machine enforcement eliminates an entire class of objective lifecycle
  bugs.

### Negative
- Polyglot adds build complexity (cargo + pip + schema sync scripts).
- Rust compile times slow iteration on kernel changes.
- JSON schemas must be manually kept in sync with Rust/pydantic types —
  drift detection is currently manual (caught via audit, not CI).

### Neutral
- The schema-first approach may slow initial development but pays off during
  integration.
- Python planner adds a runtime dependency on Python 3.11+ for the full stack
  to function.
